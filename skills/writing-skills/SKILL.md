# Writing Skills

Turn a good workflow insight into a compact, triggerable `paperpowers` skill.

## Hard Gate

Do not add or revise a `paperpowers` skill without checking when it should trigger, what behavior it should constrain, and how you will verify that it stays focused.

## Checklist

1. State the exact research workflow problem the skill should solve
2. Define the trigger conditions in one clear sentence
3. Decide the smallest workflow the skill needs to teach
4. Write the skill with explicit gates, checklist, outputs, and success condition
5. Check that the skill teaches reusable process rather than one project story
6. Verify that the description says when to use the skill, not how it works
7. Verify that the body is concise enough to be loaded repeatedly
8. Test the skill against one realistic usage scenario before declaring it ready

## Required Output

Every skill-writing pass should include:

- target workflow problem
- trigger sentence
- final skill text
- verification result
- any remaining ambiguity or trigger risk

## Key Rules

- Skills are reusable workflow guides, not project diaries
- The description should optimize discovery, not summarize the whole procedure
- Keep scope narrow enough that the skill has one dominant job
- Prefer compact wording over exhaustive prose
- If a rule could be automated mechanically, do not waste the skill on it
- If you cannot explain how to verify the skill, it is not ready

## Success Condition

The repo gains a skill that is easy to trigger, hard to misuse, and specific enough to improve future `paperpowers` behavior without bloating the system.
